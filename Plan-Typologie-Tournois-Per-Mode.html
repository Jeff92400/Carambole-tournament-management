Typologie des tournois par mode de jeu

Permettre à chaque CDB de définir une typologie de tournois (T1/T2/T3/FINALE…) différente selon le mode de jeu, avec un mécanisme d'héritage pour préserver la simplicité du cas standard.

Cible : V3.0 — release inter-saison 2026 (juin-août 2026)

Date de rédaction : 15 mai 2026

Initiative parent : Quilles (Initiatives/Quilles/) — déclencheur direct, mais bénéfice général

Pré-requis : aucun (peut être livré indépendamment du bootstrap Quilles)

1. Contexte & motivation

Aujourd'hui, la table tournament_types stocke la typologie des tournois (T1, T2, T3, FINALE…) avec une clé unique (tournament_number, organization_id). Cela signifie qu'un CDB ne peut avoir qu'une seule typologie, appliquée uniformément à tous ses modes de jeu (Libre, Cadre, Bande, 3 Bandes — et bientôt 5Q, 9Q).

Cette uniformité ne tient plus dès que :

  • Un CDB ajoute la discipline Quilles (modes 5Q / 9Q), qui pourrait fonctionner avec un nombre de phases différent du carambole.
  • Un CDB veut organiser 4 tournois + 1 finale pour Cadre mais conserver 3 tournois + 1 finale pour Libre, Bande et 3 Bandes.
  • Plus généralement, n'importe quelle évolution de format propre à un mode (ex. nombre de qualificatifs, plusieurs finales).
Décision de scope (validée en discussion) : la granularité retenue est le mode de jeu, pas la sous-catégorie (ex. Cadre 47/2 R3 vs Cadre 47/2 R4). Au sein d'un mode, toutes les catégories partagent la typologie. Le schéma intègre néanmoins un crochet category_id NULL pour permettre une descente future au niveau catégorie sans rupture.

2. Objectif

  • Flexibilité maximale : chaque mode peut avoir sa typologie propre.
  • Zéro friction pour les CDBs actuels : la typologie globale existante continue de s'appliquer par défaut à tous les modes. Aucune action requise pour les comités qui n'ont pas besoin de différencier.
  • Découvrabilité : l'admin doit pouvoir voir d'un coup d'œil quels modes ont des surcharges, sans devoir naviguer mode par mode.
  • Réversibilité : une surcharge peut être supprimée pour revenir à l'héritage global, à tout moment.

3. Modèle conceptuel : héritage à deux niveaux

Chaque ligne de typologie est rattachée à un mode de jeu, ou à aucun (auquel cas elle s'applique à tous les modes par défaut).

Lookup pour "type de tournoi numéro N, mode M, organisation O" :

  1. Chercher une ligne (N, M, O)        ← surcharge spécifique au mode
  2. Sinon, chercher (N, NULL, O)        ← typologie globale de l'organisation
  3. Sinon, valeur par défaut platform   ← cas extrême, ne devrait pas arriver

Un cas particulier : la suppression d'un type pour un mode donné (ex. 5Q n'a pas de T3 alors que la typologie globale en a un). On ne peut pas simplement "ne pas avoir de ligne", car la fallback ramènerait la ligne globale. Solution : une ligne tombstone avec is_deleted = TRUE qui indique "ce numéro n'existe pas pour ce mode".

4. Modèle de données

Schéma modifié

ALTER TABLE tournament_types
  ADD COLUMN game_mode_id INTEGER NULL REFERENCES game_modes(id) ON DELETE CASCADE,
  ADD COLUMN category_id  INTEGER NULL REFERENCES categories(id) ON DELETE CASCADE,
  ADD COLUMN is_deleted   BOOLEAN NOT NULL DEFAULT FALSE;

-- Ancienne contrainte UNIQUE à supprimer :
ALTER TABLE tournament_types DROP CONSTRAINT IF EXISTS tournament_types_unique;

-- Nouvelle contrainte UNIQUE (NULLS NOT DISTINCT pour traiter NULL comme une valeur) :
ALTER TABLE tournament_types
  ADD CONSTRAINT tournament_types_unique
  UNIQUE NULLS NOT DISTINCT (tournament_number, game_mode_id, category_id, organization_id);

CREATE INDEX idx_tournament_types_lookup
  ON tournament_types (organization_id, game_mode_id, tournament_number);
Compatibilité ascendante : toutes les lignes existantes sont migrées avec game_mode_id = NULL et category_id = NULL → comportement identique à aujourd'hui, zéro impact sur les CDBs en production.

Note sur category_id

La colonne est créée mais non exposée dans l'UI en V3.0. La logique de lookup ignore category_id tant qu'il vaut NULL. C'est une garantie de forward-compatibility : si un jour un CDB demande de différencier au niveau sous-catégorie, on étend la fallback chain sans migration de schéma.

5. Interface utilisateur

Principe : "Tous les modes" par défaut, surcharge à la demande

L'accordéon "Typologie des tournois" (Settings → Compétitions, frontend/settings-admin.html lignes 1913-1967 actuelles) gagne un sélecteur en haut :

Appliqué à : [ Tous les modes ▾ ]
              ───────────────────
              ▸ Tous les modes  ← défaut
              ▸ Carambole
                  Libre
                  Cadre
                  Bande
                  3 Bandes
              ▸ Quilles            ← visible uniquement si modes Quilles activés
                  5Q
                  9Q

Cas 1 — "Tous les modes" sélectionné (état par défaut, 99% des CDBs)

L'interface est strictement identique à aujourd'hui. Le tableau liste les types globaux. Toute édition modifie la typologie partagée.

Un indicateur discret apparaît sur les lignes qui ont des surcharges ailleurs :

N°  Code     Nom                       Classement  Finale
1   T1       Tournoi Qualificatif 1    ✓                      ⚠ surchargé pour : 5Q, 9Q
2   T2       Tournoi Qualificatif 2    ✓
3   T3       Tournoi Qualificatif 3    ✓                      ⚠ surchargé pour : 5Q (supprimé)
4   FINALE   Finale Départementale                 ✓

Cliquer sur l'avertissement navigue vers la vue du mode concerné.

Cas 2 — un mode spécifique sélectionné (ex. "5Q")

Les lignes héritées de "Tous les modes" sont affichées en grisé, avec un bouton Personnaliser par ligne :

Typologie pour 5Q   (héritée de "Tous les modes" sauf indication contraire)

N°  Code     Nom                       Classement  Finale
1   T1       Tournoi Qualificatif 1    ✓                      [Personnaliser]
2   T2       Tournoi Qualificatif 2    ✓                      [Personnaliser]
3   T3       Tournoi Qualificatif 3    ✓                      [Personnaliser]
4   FINALE   Finale Départementale                 ✓          [Personnaliser]

[+ Ajouter un type spécifique à 5Q]

Cliquer Personnaliser sur une ligne héritée crée une surcharge pour ce mode. La ligne devient pleinement éditable, visuellement distincte (bordure colorée à gauche), et un bouton Revenir au défaut apparaît.

Cas 3 — surcharges et types propres au mode

N°  Code     Nom                       Classement  Finale
1   T1       Tournoi 5Q n°1            ✓                      [⚙ surchargé]    [Revenir au défaut]
2   T2       Tournoi 5Q n°2            ✓                      [⚙ surchargé]    [Revenir au défaut]

Visibilité conditionnelle du sélecteur

Le sélecteur "Appliqué à" n'apparaît que si le CDB a au moins un mode actif au-delà du jeu de base. Pour un CDB ne pratiquant que le carambole standard, l'UI reste identique à aujourd'hui — le sélecteur est masqué.

8. Migration & compatibilité

  1. Migration SQL : ajout des colonnes, NULL par défaut sur les lignes existantes.
  2. Lecture : la fallback chain garantit que les CDBs actuels (sans surcharges) ont un comportement strictement identique.
  3. Écriture : l'UI n'expose le sélecteur que si plusieurs modes sont activés — pour un CDB carambole standard, rien ne change visuellement.
  4. Rollback : les colonnes ajoutées sont nullable, supprimer la feature consiste à ignorer les lignes non-NULL côté lookup. Aucune perte de donnée.

10. Questions ouvertes

  • Action "Revenir au défaut" sur une ligne tombstone : suppression simple, mais que faire si le mode n'a aucune surcharge restante ? Doit-on supprimer automatiquement les autres lignes orphelines du mode ? (Probablement non — laisser explicite.)
  • Action "Promouvoir vers Tous les modes" : utile si toutes les surcharges d'une ligne convergent vers la même valeur. Optionnelle pour V3.0 ?
  • Affichage du sélecteur : seuil exact de visibilité — 1 mode au-delà du carambole ? Configurable ? (Recommandation : afficher dès qu'au moins un mode Quilles est activé, sinon masquer.)
  • Granularité catégorie : confirmer définitivement qu'on ne livre pas l'UI catégorie en V3.0. Le crochet schéma reste, l'UI attend une demande terrain.
