📅 Échéance : Avant juin 2026
Décider de la stratégie URL pour la multi-CDB et du modèle de paiement pour l'hébergement et les domaines personnalisés.
Situation actuelle :
  • Gestion Tournois : cdbhs-tournament-management-production.up.railway.app (4 admins)
  • Espace Joueur : cdbhs-player-app-production.up.railway.app (57 PWA installées)
  • Paiement actuel : Personnel (propriétaire)
Stratégie recommandée :
  • Espace Joueur : Garder URL CDBHS + ajouter domaine neutre en alias (ex: joueurs-carambole.app)

Modèles de paiement possibles :

  • Hybride : URLs .up.railway.app gratuites, domaines personnalisés payants optionnels
Action requise : Réévaluer structure coûts et modèle paiement hors-saison (juillet-août 2026) avant scaling vers d'autres CDBs.

Centraliser toutes les variables de templates email dans une source unique de vérité. Actuellement, les variables sont dispersées dans 5+ fichiers, causant des problèmes de maintenance et des incohérences.
Problème actuel :
  • buildUniversalVariables() dans email.js définit certaines variables
  • Chaque endpoint email ajoute ses propres variables spécifiques
  • Le HTML frontend code en dur les listes de variables (8+ endroits)
  • CLAUDE.md documente les variables séparément
  • Ajouter une nouvelle variable nécessite des changements dans 5+ fichiers
Solution proposée :
  • Créer backend/utils/email-variables.js avec source unique
  • Ajouter endpoint API GET /api/email/available-variables
  • Frontend charge les variables dynamiquement depuis l'API
  • Toutes les fonctions d'envoi utilisent replaceAllVariables() centralisée
  • Optionnel : Stocker dans table email_template_variables pour descriptions éditables par admin
Fichiers à refactoriser :
  • backend/routes/email.js
  • backend/routes/emailing.js

Génération Calendrier - Détection Couleurs À FAIRE

⏱️ Effort estimé : 3-4 jours
📅 Échéance : Avant sept 2026
Implémenter la détection automatique des clubs via les couleurs de cellules Excel. Actuellement, le parser fonctionne mais les emplacements sont "Non défini". Objectif : 100% précision avant la saison 2026-27.
Stratégie : couleur d'abord, code lettre en fallback
  • Détection couleur (primaire) : Excel encode le club via couleur fond cellule. ExcelJS lit cell.fill.fgColor.argb
  • Code lettre (fallback) : Si aucune couleur reconnue, utiliser codes lettres (T1/A → club A, T2/B → club B)
  • Mapping CDBHS : FF974806 = Bois-Colombes, FF00B0F0 = Châtillon, etc.
Flux utilisateur (assistant 4 étapes) :
  • Étape 1 : Upload - Sélectionner saison + fichier Excel
  • Étape 2 : Validation couleurs - Parser Excel, extraire couleurs uniques, afficher mapping proposé couleur→club. Admin valide ou corrige. Mappings précédents pré-remplis.
  • Étape 3 : Aperçu - Table complète tournois avec dates, modes, catégories, emplacements (résolus)
  • Étape 4 : Import - Créer/MAJ entrées tournoi_ext
Règles validation :
  • Si cellule tournoi a couleur non reconnue ET pas de code lettre → ARRÊTER, montrer cellules non résolues
  • Finales (FD) exemptées - emplacement TBD acceptable
  • Ne jamais générer tournois avec emplacements manquants silencieusement
Implémentation :
  • Nouvelle table ou entrées organization_settings : calendar_color_mapping (JSON)
  • Endpoint backend étape 2 : POST /calendar/import-season/detect-colors
  • Frontend étape 2 : carrés colorés avec dropdown clubs, sauvegarder mapping à confirmation
  • Option auto-détection : parser zone légende Excel pour suggérer mappings

Chatbot & Chat en Direct - Espace Joueur PRÊT

⏱️ Effort estimé : 11-12 heures
📅 Planifié : 15 juin 2026
💰 Coût : < 5 €/mois
Système de chat hybride pour l'Espace Joueur - chatbot IA (Claude Haiku 4.5) pour réponses autonomes + chat direct admin pour support humain. Architecture basée polling (REST, 5s quand ouvert). Pas de WebSocket pour v1.
Flux conversation :
  • Joueur ouvre chat → conversation créée en mode ai
  • IA répond avec contexte joueur (tournois à venir, inscriptions, classements, infos org)
  • Joueur clique "Parler à un administrateur" → statut devient waiting
  • Admin voit badge notification dans navbar App Tournois → prend en charge conversation
  • Les deux côtés pollent pour nouveaux messages
  • Admin ferme conversation quand résolue
Phases :
  • Phase 1 (~30min) : Tables base de données + migrations
  • Phase 2 (~4-5h) : Chatbot IA + frontend Espace Joueur
  • Phase 3 (~4-5h) : Support admin + interface App Tournois
  • Phase 4 (~2h) : Paramètres & finitions
Prérequis : Clé API Anthropic, ANTHROPIC_API_KEY sur Railway. Coût : < 5 EUR/mois avec Haiku 4.5. Décisions encore ouvertes :
  • Durée rétention conversation (avant suppression) ?
  • Plusieurs admins peuvent répondre ou un par CDB ?
  • Notification email à admin quand joueur demande support humain ?

Analytiques Historiques Joueur (Stats Multi-Saisons) CONCEPT

📅 Échéance : Aucune
⚙️ Changements schéma : Aucun requis
Le modèle de données supporte déjà ceci. Toutes les données tournois (tournament_results, rankings) sont conservées de façon permanente entre saisons et scopées par organization_id. Possibilité de construire analytiques riches joueurs au fil du temps.
Analytiques possibles :
  • Progression moyenne par mode entre saisons
  • Tendances participation (tournois joués par saison, régularité)
  • Évolution classement (position en catégorie année après année)
  • Progression meilleure série
  • Stats cross-CDB au niveau Ligue ou FFB (toutes orgs partagent même base)
Volume données à l'échelle (90 CDBs + 16 Ligues) : < 1M lignes après 10 ans sur toutes tables - pas de souci performance ou stockage pour PostgreSQL. Requêtes indexées par season et organization_id. Implémentation : Uniquement requêtes + UI (probablement dans onglet stats Espace Joueur). Pas de changements schéma nécessaires.
