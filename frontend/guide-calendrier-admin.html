1. Vue d'ensemble du moteur

L'Assistant Calendrier remplace la construction Excel manuelle d'un calendrier CDB. Le moteur place automatiquement tous les tournois (T1, T2, T3, Finale) de toutes les catégories actives sur les week-ends éligibles, en respectant les contraintes définies par l'admin.

Algorithme

  • Greedy round-by-round : tous les T1 sont placés en premier (toutes catégories), puis les T2, puis les T3, puis les Finales.
  • Pour chaque (catégorie, type), le moteur teste tous les week-ends candidats, élimine ceux qui violent une règle dure, puis choisit celui qui minimise le score des règles souhaitées.
  • Si une étape de chaîne échoue pour une catégorie (ex. T2 non plaçable), les étapes suivantes (T3, Finale) sont automatiquement marquées « chaîne interrompue » plutôt que produire un calendrier incohérent.

Limites connues

  • Pas de backtracking : le moteur ne revient pas sur le placement d'un T1 si un T2 ultérieur échoue. Workaround : verrouillage manuel d'un T1 sur la date qui marche.
  • Pas de capacité par club : un club avec 4 billards et un avec 1 billard sont traités à égalité pour l'équilibre des charges.
  • Affinités cross-mode heuristiques : déduites du niveau (R1↔R1, R3↔R4) et non d'une matrice spécifique au CDB.

3. Réglages avancés (panneau admin)

Le bouton « 🔧 Réglages avancés (admin) » sous l'encart vert ouvre la bibliothèque complète de règles. C'est l'outil principal pour ajuster finement le comportement du moteur.

Tables affichées

  • Règles obligatoires : le moteur refuse de les violer. Paramètre éditable (ex. min_weeks).
  • Règles souhaitées : optimisées via un poids modifiable (0 = ignorée, 5 = léger, 10 = moyen, 50+ = dominant).
  • Règles disponibles : règles non encore activées du catalogue. Boutons « Ajouter » ou « Ignorer ».

Sauvegarde automatique

Chaque modification (paramètre, poids, activation, ajout, suppression) est persistée immédiatement en base. Aucun bouton « Sauvegarder » n'est requis. Pour appliquer aux placements, lancer « ⚡ Tout régénérer ».

5. Catalogue des règles souhaitées

Chaque règle souhaitée ajoute un score (pénalité) au créneau candidat. Le moteur choisit le score minimal.

RèglePoids défautEffet
Étalement entre week-ends (anti-clusters)5Pénalise les WE déjà chargés (quadratique : n²). Pousse à étaler.
Nombre équilibré de tournois par club hôte5Pénalise les clubs déjà très servis (linéaire). Vise une répartition équitable.
Pas d'hôte deux week-ends consécutifs3Pénalité fixe (w × 10) si le club a un tournoi le WE −1 ou +1.
Étalement par mode + affinités de niveau5Pénalise les paires de catégories corrélées sur même/adjacent WE :
  • Même mode : facteur 1.0
  • Modes différents, même niveau (R1↔R1) : facteur 0.8
  • Modes différents, niveau adjacent : facteur 0.4
Charge équilibrée entre les mois10Pénalise les mois sur-remplis (quadratique, normalisé par capacité). La règle la plus impactante chez la plupart des CDB.
Respect des dates cibles T2/T3/Finale (rétroplanning)10Tire chaque tournoi vers sa date cible calculée par rétroplanning depuis la Finale Ligue (ou la fin de saison). Cadre la cascade.
Règle masquée : t1_earliness_weight est conservée en base pour rétro-compatibilité mais non affichée dans l'UI depuis V 2.0.686. Elle est redondante avec « Respect des dates cibles ».

6. Comment fonctionne le scoring

Pour chaque créneau candidat (date + club hôte), le moteur calcule un score additif :

  1. Toutes les règles dures sont vérifiées en premier. Un échec = rejet immédiat avec une raison stockée pour le diagnostic.
  2. Si toutes les règles dures passent, le moteur évalue chaque règle souhaitée et accumule sa contribution dans le score (pénalité × poids).
  3. Le créneau au score le plus bas est retenu. En cas d'égalité, le tiebreaker est la date la plus précoce, puis l'ID de catégorie pour le déterminisme.

Lire les diagnostics

Si un tournoi n'est pas placé, l'encart conflits affiche par exemple :

⚠️ Bande N3 Finale — 52 créneaux testés, tous rejetés : 14× trop tard avant la finale ligue · 12× après le dernier WE de compétition · 8× Finale placée avant T3 · 5× WE blackout · 5× < 3 sem. depuis T2 · 3× < 3 sem. depuis T3

Chaque ligne indique combien de WE ont été rejetés pour quelle raison. La somme totale = 52 (tous les WE testés). Pour réparer un conflit :

  • Si le diagnostic est dominé par « trop tard avant la finale ligue » → la deadline ligue est trop serrée. Vérifier la date saisie en Étape 1, ou réduire min_weeks_between_cdb_and_ligue_final.
  • Si dominé par « < N sem. depuis T2/T3 » → la chaîne précédente s'est terminée trop tard. Verrouillage manuel d'un T1/T2 plus tôt.
  • Si dominé par « max atteint » → assouplir max_tournaments_per_weekend ou activer/renforcer l'anti-cluster.

7. Publication sur la page Calendrier admin

Le bouton « 📤 Publier sur la page Calendrier » pousse les placements du draft vers la table publique tournoi_ext. Conséquences :

  • Les tournois deviennent visibles aux joueurs via la page Calendrier et l'Application Joueur.
  • Les inscriptions s'ouvrent (si la date d'ouverture est passée).
  • Les emails de relance et notifications push deviennent possibles pour ces tournois.
Action engageante. À n'effectuer qu'après validation finale par le bureau du CDB. La modification ultérieure d'une date ou d'un club est possible mais déclenche des notifications aux joueurs déjà inscrits (changement de date / lieu / annulation).

Sync re-publication

Si vous re-publiez après une régénération partielle, le moteur fait un diff entre le draft et la table tournoi_ext :

  • Nouveaux placements : ajoutés.
  • Dates/clubs modifiés : mis à jour, avec push notification aux inscrits si applicable.
  • Tournois retirés du draft : non supprimés automatiquement (trace conservée). Utiliser cleanup-externals si besoin.

8. Dépannage : conflits non résolus

Symptôme : « X / Y placés », X < Y

Cause probableAction
Saison trop chargée + max/WE trop basAugmenter max_tournaments_per_weekend à 3 ou 4.
Trop de blackouts en début de saisonRéduire le nombre de blackouts ou avancer first_weekend.
Finale Ligue très tôt + cascade T1 longueVerrouiller manuellement T1 de la catégorie problématique sur un WE précoce.
Indisponibilité club rendant la cascade infaisableÉlargir la liste des clubs hôtes confirmés en Étape 1.

Symptôme : « 11 conflits » (massif)

Régression engine probable. Vérifier :

  • Les défauts du catalogue n'ont pas été modifiés par erreur (ex. category_cadence_weight à 0 ou 50 au lieu de 10).
  • Les règles essentielles sont bien activées.
  • Si le problème persiste après régénération, contacter le développeur.

Symptôme : T3 sans T2 (ne devrait plus arriver)

Depuis V 2.0.685, le moteur saute T3/Finale si T2 a échoué et reporte « chaîne interrompue ». Si vous voyez encore un T3 orphelin, c'est un bug à signaler.

9. Cas d'usage avancés

Verrouillage stratégique avant régénération

Exemple : mi-mars, la finale Bande R1 est annoncée pour le 15 mai, convocations envoyées. Le 1er mai, la fédération annonce un changement de date pour la finale Ligue Bande R1.
  1. Étape 3 : cliquer chaque cellule déjà annoncée → 🔒 verrouiller (commentaire optionnel).
  2. Étape 1 : mettre à jour la date de la finale Ligue Bande R1.
  3. Étape 3 : cliquer « 🔒 Régénérer en gardant mes verrous ».
  4. Le moteur préserve les cellules verrouillées et ajuste uniquement les Finales/T3 impactés.

Importer un calendrier existant pour démarrage

Pour les CDB qui n'ont pas encore basculé sur le moteur, deux approches :

  • Démarrage à zéro : créer un brief vierge pour la saison N+1, charger les règles habituelles, générer.
  • Import depuis tournoi_ext existant : si la saison courante a déjà des tournois publiés, ils apparaissent en lecture seule dans la vue « Saison publiée » de l'Étape 3. Servent de référence visuelle.

Plusieurs cycles de régénération en cours de saison

Pattern courant : régénérer à chaque modification structurelle (changement de Ligue Final, indispo nouvelle d'un club, blackout ajouté). Toujours « 🔒 Régénérer en gardant mes verrous » pour préserver les engagements joueurs déjà pris.

10. Bonnes pratiques

  • Saisir l'Étape 1 en premier et complètement : 80 % des conflits viennent d'une donnée brief manquante (Ligue Final non saisie, blackout oublié, club mal coché).
  • Garder les 3 réglages essentiels par défaut sauf situation atypique. Les valeurs validées : max=3, espacement=3, respiration ligue=3.
  • Régénérer souvent : le moteur est rapide (< 2 s). Tester plusieurs configurations.
  • Verrouiller avant de communiquer aux joueurs ou aux clubs. Une fois verrouillé, le moteur respecte le placement.
  • Publier en dernier, après validation officielle. Re-publier après chaque modification majeure pour synchroniser tournoi_ext.
  • Email aux clubs hôtes en mode preview avant l'envoi réel — vérifier le contenu personnalisé.
Conseil pour les nouveaux CDB : avant la première saison, faire 2–3 régénérations test avec des valeurs extrêmes (max=2 puis max=5) pour comprendre l'impact des règles. Cela donne une intuition pour les saisons suivantes.
